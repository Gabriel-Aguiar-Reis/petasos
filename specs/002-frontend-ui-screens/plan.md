# Implementation Plan: Frontend UI Screens — RoadLedger Mobile App

**Branch**: `002-frontend-ui-screens` | **Date**: 2026-04-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-frontend-ui-screens/spec.md`

## Summary

Build the complete mobile UI for RoadLedger: 8 screens (Dashboard, Trips, Costs, Fuel, Work Sessions, Goals, Export, More hub), a persistent FAB quick-entry overlay, React Query hooks bridging domain use cases to screens, two Zustand stores for UI state, and all UI components using NativeWind + React Native Reusables (@rn-primitives). The `001-app-backend` domain layer is consumed as-is without modification.

## Technical Context

**Language/Version**: TypeScript 6.0.2 — strict mode, no `any`  
**Primary Dependencies**: React Native 0.85.1, Expo SDK 55.0.15, Expo Router 55.0.12, NativeWind 4.2.3, @rn-primitives 1.4.0, lucide-react-native 1.8.0, class-variance-authority 0.7.1, clsx 2.1.1, tailwind-merge 3.5.0, expo-file-system 55.0.16, expo-sharing 55.0.18  
**Storage**: SQLite via expo-sqlite + Drizzle ORM (from `001-app-backend` — no changes)  
**Testing**: Jest 29.7.0 + @testing-library/react-native 13.3.3 — component tests for hooks; no E2E in MVP  
**Target Platform**: Android only (MVP) — Expo managed workflow  
**Project Type**: Mobile app (React Native / Expo)  
**Performance Goals**: Dashboard loads in < 500 ms for 300 trips + 150 costs + 30 fuel logs (SC-002); JSON export opens share sheet in < 3 s for 30 days of data (SC-005)  
**Constraints**: 100% offline; no network calls in screen or hook layer; domain layer untouched; tap targets ≥ 44×44 dp; key actions in ≤ 3 taps (SC-001, SC-006)  
**Scale/Scope**: Single-user, single-device MVP; 8 screens, ~20 components, 10 React Query hooks, 2 Zustand stores

## Constitution Check

_GATE: Must pass before Phase 0 research. Re-check after Phase 1 design._

| Principle                                 | Status  | Notes                                                                                                                                                                |
| ----------------------------------------- | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| I. Offline-First                          | ✅ PASS | All data reads/writes go through domain use cases → SQLite. No network calls in hooks or screens. `expo-sharing` uses the device's local share sheet — no upload.    |
| II. Driver-Centric UX                     | ✅ PASS | FAB quick-entry requires ≤ 2 fields (earnings+platform or amount+category). Tap targets enforced at ≥ 44 dp. Dark mode is default. One-hand tab navigation (5 tabs). |
| III. Spec-Driven Development              | ✅ PASS | Plan derives directly from the approved `spec.md` for `002-frontend-ui-screens`.                                                                                     |
| IV. Clean Architecture & Domain Isolation | ✅ PASS | Screens → hooks → use cases → repositories. Domain entities/use cases untouched. Drizzle repositories do not leak into screens.                                      |
| V. Platform Agnosticism                   | ✅ PASS | Platform field is free-form string in all forms; no platform-specific logic in UI code.                                                                              |
| VI. Data Ownership & Exportability        | ✅ PASS | Export screen invokes `ExportDataAsJSON` and shares via native share sheet. JSON MVP export is fully implemented.                                                    |
| VII. Simplicity & YAGNI                   | ✅ PASS | No charts (Victory Native — Post-MVP), no GPS, no notification parsing, no XLSX. Victory Native and expo-location not installed in this feature.                     |

**Post-design re-check**: All principles continue to pass. No violations to track.
