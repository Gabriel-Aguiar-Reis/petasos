# Research: App Backend — Clean Architecture Domain & Data Layer

**Feature**: `001-app-backend`
**Date**: 2026-04-15
**Status**: Complete — all NEEDS CLARIFICATION resolved

## R-001: Expo SDK & React Native Version

**Decision**: Expo SDK **55.0.15** + React Native **0.85.1** + TypeScript **6.0.2** (strict)

**Rationale**: SDK 55 is the current stable release as of April 2026. It ships React Native 0.85 which includes the New Architecture (Fabric + TurboModules) enabled by default. `expo-sqlite` in SDK 55 exposes the synchronous and async APIs needed by Drizzle ORM. Expo Router **55.0.12** is bundled with SDK 55 for file-based navigation.

**Alternatives considered**: SDK 54 — rejected because SDK 55 LTS is available and the New Architecture stabilises SQLite performance on Android.

---

## R-002: Drizzle ORM + expo-sqlite Integration

**Decision**: `drizzle-orm` **0.45.2** + `expo-sqlite` **55.0.15** via the `drizzle-orm/expo-sqlite` adapter. `drizzle-kit` **0.31.10** for migration generation. Migrations applied at app startup via Drizzle's `migrate()` helper with bundled migration files generated by `drizzle-kit`.

**Rationale**:

- `expo-sqlite` in SDK 55 provides `openDatabaseSync` which Drizzle's Expo adapter wraps natively.
- `drizzle-kit generate` produces versioned SQL migration files; `migrate(db, migrations)` applies them on boot — zero network, purely local.
- Type inference from schema eliminates runtime casting in repository implementations.

**Key conventions**:

- Schema files live in `src/infra/db/schema/` — one file per entity.
- Migration output: `src/infra/db/migrations/` (auto-generated, committed to repo).
- DB singleton opened once in `src/infra/db/client.ts` and injected into repositories.

**Alternatives considered**: WatermelonDB — rejected (heavier, less type-safe, worse Drizzle synergy). SQLite directly via `expo-sqlite` without ORM — rejected (no type safety, verbose query composition).

## R-003: Clean Architecture Folder Structure for Expo/React Native

**Decision**:

```
src/
├── app/                    # Screens (Expo Router)
├── application/
│   ├── use-cases/          # Application logic
│   ├── hooks/              # React Query hooks
│   └── services/           # Application services
├── domain/
│   ├── entities/           # Business entities
│   ├── repositories/       # Repository interfaces
│   └── validations/        # Validation schemas (Zod)
├── infra/
│   ├── db/                 # Database configuration
│   └── repositories/       # Repository implementations (Drizzle)
├── components/
│   └── ui/                 # UI components
├── lib/                    # Utilities and helpers
└── types/                  # TypeScript type definitions
tests/                      # Jest test files
```

**Rationale**: Domain layer has zero imports from `infra/` or `app/`. Use cases accept repository interfaces — implementations injected at the composition root (`app/` layer). This satisfies Constitution Principle IV and SC-001 (pure in-memory stubs for testing).

**Alternatives considered**: Feature-folder structure (e.g., `features/trips/domain/`) — rejected for MVP because the entity graph is small and cross-cutting; flat domain is simpler (Constitution Principle VII).

## R-004: Testing Strategy

**Decision**: Jest **29.7.0** + ts-jest **29.4.9** + `@testing-library/react-native` **13.3.3** for unit and integration tests. Domain use cases tested with pure in-memory repository stubs (zero device/app dependency, satisfies SC-001). No E2E framework in MVP.

**Rationale**: Domain entities and use cases are plain TypeScript — Jest runs them in Node without any React Native bridge. In-memory repositories implement the domain interfaces with `Map<string, Entity>` stores, making test setup trivial. Detox (E2E) deferred to Post-MVP per Constitution Principle VII.

**Version pin rationale**: Jest 30 is the latest major but `ts-jest` does not yet support it (ts-jest latest is 29.4.9). Pinning both to the 29.x series ensures compatibility. Re-evaluate when ts-jest releases a Jest 30-compatible version.

**Test file conventions**:

- Unit tests: co-located `__tests__/` folder beside source file.
- Integration tests (use case + in-memory repo): `tests/` at project root.

**Alternatives considered**: Vitest — compatible but Jest is the default for Expo projects; switching adds friction without benefit. `@swc/jest` as transformer — faster but adds a native binary dependency; ts-jest is simpler for this project size.

## R-005: State Management Integration (Zustand + TanStack Query + Zod)

**Decision**:

- **TanStack Query** `@tanstack/react-query` **5.99.0** (`useQuery`, `useMutation`) wraps all use case calls. Treats SQLite as the "server" — no network fetching.
- **Zustand** **5.0.12** manages pure UI state: overlay open/closed, form draft values, active tab.
- **Zod** **4.3.6** validates all use case inputs at the domain boundary. Note: Zod v4 changed the API — top-level helpers (`z.string()`, `z.number()`, etc.) remain; UUID validation is now `z.string().uuid()` (unchanged in v4).
- `queryClient.invalidateQueries()` called in mutation `onSuccess` to keep Query cache fresh after writes.

**Rationale**: TanStack Query provides caching, loading/error states, and background refetch out of the box — appropriate even for local SQLite because the async read cost is non-trivial on large datasets. Zustand is lightweight and suitable for ephemeral UI state. Zod provides runtime safety at use-case input boundaries without coupling to Expo/React Native.

**Key convention**: Use case calls are always `async` (even if SQLite op is synchronous) to keep the query boundary clean. Repository implementations wrap synchronous Drizzle calls in resolved promises. Zod schemas live in `src/domain/validations/` and are imported only by use cases — never by entities or repository interfaces.

## R-006: Error Handling Strategy

**Decision**: Four typed error classes extending a base `AppError`:

- `ValidationError` — invalid input (thrown by use cases before any repository call)
- `StorageError` — SQLite read/write failure (thrown by repository implementations, wraps the raw error)
- `NotFoundError` — entity not found in `findById`, `update`, or `delete`
- `ConflictError` — business rule violation (e.g., `StartWorkSession` when session active)

All are serialisable (include `code: string` + `message: string`) for TanStack Query error boundaries.

**Alternatives considered**: Discriminated union Result type (`Result<T, E>`) — more functional but verbose in TypeScript without a dedicated library; rejected per Constitution Principle VII.

## R-007: FuelType Handling

**Decision**: `fuelType` is a required `TEXT` column on `fuel_logs`. No enum enforced at the DB level — stored as free-form string. Standard suggested values ("Gasolina", "Etanol", "Diesel", "GNV") provided as domain constants for UI autocomplete, but the column accepts any non-empty string.

**Rationale**: Brazilian market has regional variants and the flex-engine use case (Gasolina/Etanol alternation) requires per-type km/L tracking. DB-level enum would require a migration for every new fuel type variant — fragile. Domain constants provide UX guidance without schema coupling.

**km/L grouping**: `GetFuelEfficiency` queries `fuel_logs` ordered by `odometer ASC`, groups by `fuelType`, and computes efficiency pair-wise within each group. Returns `FuelEfficiencyResult[]` where each item has `{ fuelType, kmPerLiter, costPerKm, logCount }`.

## R-008: Quick-Entry Mode for CreateTrip and CreateCost

**Decision**: Single use case, single input type — optional fields marked `undefined` (not overloaded).

- `CreateTripInput`: `{ earnings: number; platform: string; date?: Date; origin?: string; destination?: string; distance?: number; duration?: number; vehicleId?: string }`
- `CreateCostInput`: `{ amount: number; category: string; date?: Date }`

**Rationale**: No separate "quick-entry mode" flag needed — the use case already validates only required fields and defaults `date` to `new Date()` when omitted. This satisfies FR-016 with no additional complexity.

## R-009: JSON Export Implementation

**Decision**: `ExportDataAsJSON` use case returns a typed `ExportEnvelope` object. The UI layer is responsible for serialising it to a file using Expo's `expo-file-system` + `expo-sharing`. The use case itself has no file system dependency (satisfies domain isolation, Constitution Principle IV).

**Export envelope type**:

```typescript
type ExportEnvelope = {
  exportedAt: string // ISO 8601
  filters: {
    dateRange?: { from: string; to: string }
    platform?: string
  }
  trips: Trip[]
  costs: Cost[]
  fuelLogs: FuelLog[]
  workSessions: WorkSession[]
  goals: Goal[]
}
```

## R-010: ID Generation

**Decision**: Use `uuid` **v13.0.0** (npm: `uuid`) instead of `nanoid` for entity ID generation.

**Rationale**: `nanoid` v5 is ESM-only and requires additional Jest configuration (transform ignore patterns) to work in a CommonJS Jest environment. `uuid` v13 ships both ESM and CJS builds, works out of the box with Jest 29 + ts-jest, and produces standard UUID v4 identifiers — identical in function to nanoid for this use case.

**Usage**: `import { v4 as uuidv4 } from 'uuid'` in domain use cases and entities. No domain knowledge of UUID format required — IDs are opaque strings at the domain boundary.
